## 1. 快速入门：5 分钟跑通第一个 TRIZ 分析

### 1.1 激活技能

在 Claude Code 对话中，直接描述你的问题即可。当你的描述匹配 TRIZ 技能的触发条件时，技能自动加载。

**最简触发语：**

```
我的电机功率密度提升时，散热跟不上，怎么办？
```

Claude 会自动加载 TRIZ 技能，并执行问题路由（Problem Routing）。

### 1.2 你会看到什么

技能加载后，Claude 会按以下流程响应：

```
1. 问题诊断    → 识别你面临的是 Technical Contradiction（技术矛盾）
2. 参数映射    → "功率密度↑" → 参数 21 (Power)
                 "散热恶化↑" → 参数 22 (Loss of Energy)
3. 矛盾矩阵查询 → 从 cheatsheet.md 查 39×39 矩阵
4. 推荐原则    → 推荐 40 Principles 中排名前 4 的原则
5. 方案生成    → 每个原则给出 2-3 个具体工程方案
6. 报告输出    → 自动保存到 report/ + 自动调用 NotebookLM 生成 PPT
```

### 1.3 主动调用方式

你也可以显式指定使用 TRIZ 技能：

```
用TRIZ分析一下：我的无人机螺旋桨噪音大但转速不能降低，怎么解决？
```

或者使用英文：

```
TRIZ analysis: how to reduce battery pack weight without sacrificing capacity?
```

## 2. 触发机制：TRIZ 技能如何被激活

TRIZ 技能的激活采用**关键词 + 语义双重匹配**机制。以下是完整的触发条件表：

### 2.1 自动触发（关键词检测）

| 触发信号 | 示例 | 路由目标 |
|---------|------|---------|
| 技术矛盾（trade-off / "A好B差"） | "功率大了但温度高了" | Ch03 矛盾矩阵 |
| 物理矛盾（自相矛盾） | "它需要既大又小" | Ch03 分离原理 |
| 下一代产品设计 | "下一代产品应该怎么做" | Ch04 技术进化趋势 |
| 降本 / 简化 | "怎么让这个东西更便宜" | Ch14 裁剪 + Ch05 理想度 |
| 卡住了 / 想不出办法 | "试过所有方法都不行" | Ch07 心理惯性突破 |
| 专利策略 | "怎么保护我们的技术路线" | Ch04 趋势 + Ch22 实施 |
| 失效预测 / 安全性 | "这个设计会怎么坏" | Ch18 AFI |
| 商业模式创新 | "商业模式怎么创新" | Ch19 商业TRIZ |
| 方案落地 | "怎么把方案变现/产业化" | Ch22 实施方法论 |

### 2.2 功能触发（结构检测）

当你的问题包含以下**结构特征**时，即使没有出现关键词，技能也会激活：

- **对比结构**："X 提高了，但 Y 下降了" → 识别为技术矛盾
- **否定+肯定**："不想增加重量，但需要更高强度" → 识别为物理矛盾
- **时序结构**："先...然后...但是现在..." → 识别为系统演进问题
- **否定列举**："试了 A 不行，B 也不行" → 识别为思维惯性

### 2.3 显式触发

你也可以直接使用以下指令强制激活：

```
/TrizInv
```

或：

```
@Triz 分析这个问题：...
```

### 2.4 触发后的路由逻辑

技能激活后，Claude 会自动执行 SKILL.md 中的 **PROBLEM ROUTING（11 个诊断问题）**，按顺序判定：

```
Q1: 问题是否清晰定义？
  ├─ NO  → Ch12 功能分析（先搞清楚系统到底是什么）
  ├─ YES → Q2
  └─ 想先定义理想目标 → Ch09 IFR

Q2: 是什么类型的矛盾？
  ├─ 技术矛盾 → 39×39 矩阵 + 40 原则
  ├─ 物理矛盾 → 分离原理（时间/空间/条件/系统级）
  └─ 不确定 → Ch03 诊断

Q3-Q11: (按需路由)
  Q3: 预测未来技术？        → Ch04
  Q4: 降本/简化？           → Ch14
  Q5: 创造受阻？            → Ch07/Ch08
  Q6: 物-场交互问题？       → Ch13
  Q7: 找实现方法？          → Ch06
  Q8: 极难且其他方法无效？   → Ch17 ARIZ
  Q9: 预测/防止失效？       → Ch18 AFI
  Q10: 商业/管理问题？      → Ch19/Ch20/Ch21
  Q11: 准备实施？           → Ch22
```

## 3. 功能判定原则：根据问题特征选择工具

这是 TRIZ 技能最核心的设计思想：**不根据你的行业选工具，而是根据你的问题特征选工具。**

### 3.1 五大问题类别判定

| 类别 | 判定特征 | 首选工具 | 次选工具 |
|------|---------|---------|---------|
| **Trade-off 类** | 两个参数此消彼长 | 矛盾矩阵 + 40 原则 | 76 标准解 |
| **Self-Contradictory 类** | 同一参数需要相反状态 | 分离原理 | ARIZ |
| **Degraded 类** | 系统性能全面变差 | 功能分析 → 裁剪 | 理想度分析 |
| **Stuck/Mental Block 类** | 无法想出任何新思路 | 心理惯性工具 | 9-Windows |
| **Failure-Prone 类** | 担心系统会出问题 | AFI | 功能分析 |

### 3.2 判定决策树（完整版）

```
你的问题
│
├─ 你知道具体要改善什么参数吗？
│   ├─ NO → [功能判定：不清晰]
│   │        → Ch12 功能分析（绘制组件-功能-交互图）
│   │        → 找到根因后回到此决策树
│   │
│   └─ YES → 改善参数 A 会导致参数 B 恶化吗？
│       ├─ YES → [功能判定：技术矛盾]
│       │        → 从 39 参数表中找到 A 和 B
│       │        → 查矛盾矩阵（cheatsheet.md 第 60-80 行）
│       │        → 应用推荐的前 4 条发明原则
│       │        → 每个原则生成 2-3 个具体方案
│       │
│       └─ NO → 参数 A 自身需要两个相反状态吗？
│           ├─ YES → [功能判定：物理矛盾]
│           │        → 分离原理（时间/空间/条件/系统）
│           │        → 如无效 → 升级到 Ch17 ARIZ
│           │
│           └─ NO → 问题涉及物-场交互吗？
│               ├─ YES → [功能判定：物-场问题]
│               │        → Su-Field 建模
│               │        → 76 标准解（古典 + Oxford）
│               │
│               └─ NO → 问题涉及未来预测吗？
│                   ├─ YES → [功能判定：趋势问题]
│                   │        → Ch04 8 大技术进化趋势
│                   │        → S-Curve 判定系统阶段
│                   │
│                   └─ NO → 问题涉及失效预防吗？
│                       └─ YES → [功能判定：安全问题]
│                                → Ch18 AFI（反转-放大-生成-再反转）
```

### 3.3 功能组合原则

大多数真实问题需要**多个工具联动**。组合规则如下：

| 场景 | 工具组合 | 顺序 |
|------|---------|------|
| 新产品概念开发 | IFR → 趋势预测 → 矛盾解决 → AFI | Ch09 → Ch04 → Ch03 → Ch18 |
| 现有产品降本 | 功能分析 → 裁剪 → 矛盾解决 | Ch12 → Ch14 → Ch03 |
| 突破性创新 | 9-Windows → 趋势预测 → ARIZ | Ch08 → Ch04 → Ch17 |
| 技术风险管理 | AFI → 功能分析 → 标准解 | Ch18 → Ch12 → Ch13 |
| 专利布局 | 趋势预测 → 矛盾矩阵 → 实施 | Ch04 → Ch03 → Ch22 |

## 4. 技能联动架构：5 个组件如何协同工作

TRIZ 技能不是平铺的 24 章文档，而是一个**分层联动**的知识系统。

### 4.1 架构总览

```
┌─────────────────────────────────────────────────────────────┐
│                      SKILL.md（入口 + 路由）                   │
│  - YAML 元数据（触发条件）                                     │
│  - Problem Routing（11 个诊断问题，决策树）                     │
│  - Quick Tool Selector（按目标查工具）                         │
│  - 5-Phase Process（完整创新流程）                             │
│  - Auto-Invocation Rules（Triz-AI 联动规则）                   │
│  - Report Generation（报告 + PPT 自动生成链）                   │
└──────────┬──────────┬──────────┬──────────┬─────────────────┘
           │          │          │          │
    ┌──────▼──┐ ┌────▼────┐ ┌───▼────┐ ┌──▼──────────┐
    │chapters/│ │patterns │ │cheat-  │ │  glossary   │
    │ 24 章   │ │  .md    │ │sheet.md│ │   .md       │
    │深度知识  │ │ 模式库  │ │ 速查表  │ │ 术语词典    │
    └─────────┘ └─────────┘ └────────┘ └─────────────┘
```

### 4.2 各组件的角色与联动

| 组件 | 角色 | 何时加载 | 与谁联动 |
|------|------|---------|---------|
| **SKILL.md** | 总调度器 | 始终（技能入口） | 路由到所有其他组件 |
| **cheatsheet.md** | 战场速查卡 | 解决矛盾时 | 为 SKILL.md 的矛盾路由提供 39 参数表、Top 20 矛盾矩阵、ARIZ/AFI 检查表 |
| **patterns.md** | 模式兵器库 | 应用具体工具时 | 为章节提供 40 原则详细子步骤、8 大趋势阶段策略、76 标准解完整分类、实施阶梯 |
| **chapters/*** | 深度知识库 | 需要完整方法论时 | 每个章节引用 patterns.md 的模式细节、cheatsheet.md 的速查表、glossary.md 的术语 |
| **glossary.md** | 术语翻译器 | 遇到陌生术语时 | 为所有其他组件提供 A-Z 定义 + 章节交叉引用 |

### 4.3 跨组件联动示例

以"电机转矩密度提升但温升恶化"为例，展示 5 个组件如何联动：

```
Step 1: SKILL.md 路由
  → Problem Routing Q2: "扭矩↑但温升↑" = Technical Contradiction
  → 参数映射: 扭矩 → 参数 21 (Power), 温升 → 参数 22 (Loss of Energy)
  → 路由到 Ch03 + cheatsheet.md

Step 2: cheatsheet.md 速查
  → 参数 21 vs 22: 推荐原则 35, 19, 2, 15
  → 原则 35 = Parameter Changes (改变参数状态)
  → 原则 19 = Periodic Action (周期性动作)
  → 原则 2 = Taking Out (提取/分离)
  → 原则 15 = Dynamism (动态化)

Step 3: patterns.md 展开原则
  → 原则 35 的子策略: 改变物理状态/浓度/柔性/温度
  → 原则 19 的子策略: 脉冲替代连续/变频/用暂停做其他事
  → 原则 15 的子策略: 可调参数/可拆分相对运动/流体→场

Step 4: chapters/ch03-40-principles.md 深度理解
  → 电机领域的经典案例
  → 反模式（哪些原则在电机领域效果差）
  → 精神模型（如何判断原则是否适用）

Step 5: 方案生成后联动 Ch18 AFI
  → 对每个方案进行失效预测
  → "如果用相变材料散热，什么情况会失效？"
  → 反转-放大-生成-再反转

Step 6: 自动报告 + PPT
  → SKILL.md 的 Report Generation 规则自动触发
  → 保存到 report/xxx.md
  → 自动调用 NotebookLM skill 生成 PPT
```

### 4.4 Tool Coverage Matrix 联动

当你对某个工具的权威性有疑问时，SKILL.md 底部的 **Tool Coverage Matrix** 告诉你每个工具在 7 本源书中的覆盖程度：

```
例如：你想用 ARIZ
→ Tool Coverage Matrix 显示:
  Dummies: ✓ (简介)
  Livotov: ✓✓✓ (全面)
  Gadd: ✓✓ (5-step)
  金钥匙: ✓ (简介)
  Altshuller: ✓✓✓ (原始来源)
  Bukhman: ✓ (简介)
→ 你应该参考 Livotov (现代应用) 和 Altshuller (原始算法)
→ 对应 chapters/ch17-ariz.md (综合)
```

## 5. 最优工作流选择：6 条路线图

根据你的角色和目标，选择对应的最优工作流。

### 5.1 路线图索引

```
你的身份 → 你的目标 → 推荐路线
```

### 路线 A：工程师解决具体问题（最常用）

**适用场景**：你在开发产品，遇到了一个明确的技术难题。

```
问题描述
  │
  ├─ 1. 矛盾识别（2 分钟）
  │   → 用 39 参数表（cheatsheet.md）映射你的问题
  │   → 判定是技术矛盾还是物理矛盾
  │
  ├─ 2. 原则匹配（3 分钟）
  │   → 查矛盾矩阵（cheatsheet.md Top 20 / 完整矩阵）
  │   → 获取推荐的前 4 条原则
  │   → 从 patterns.md 查看每条原则的子策略
  │
  ├─ 3. 方案生成（5 分钟）
  │   → 逐条原则生成 2-3 个具体工程方案
  │   → 用 Ch09 IFR 评估方案理想度
  │   → 选出 Top 3 方案
  │
  ├─ 4. 失效检查（3 分钟）
  │   → 用 Ch18 AFI（反转法）对 Top 3 方案做失效预测
  │   → 提前埋入防护措施
  │
  └─ 5. 输出（自动）
      → 报告保存到 report/
      → PPT 自动生成
```

**总耗时**：约 15 分钟
**涉及组件**：SKILL.md → cheatsheet.md → patterns.md → Ch03 → Ch09 → Ch18

### 路线 B：技术管理者做产品路线图

**适用场景**：你是 CTO/技术总监，需要规划下一代产品。

```
战略目标
  │
  ├─ 1. 当前系统评估（5 分钟）
  │   → Ch04 S-Curve 分析：你的产品在哪个阶段？
  │   → Ch05 理想度 = Benefits / (Costs + Harms)
  │   → Ch12 功能分析：当前系统的瓶颈在哪？
  │
  ├─ 2. 未来趋势预测（5 分钟）
  │   → 8 大技术进化趋势（patterns.md Part 2）
  │   → 每条趋势：你的系统当前处于什么位置？
  │   → 识别 3-5 个进化方向
  │
  ├─ 3. 机会空间识别（5 分钟）
  │   → 自动触发 triz-ai discover --domain "your_domain"
  │   → 发现该领域专利白空间（未充分利用的原则）
  │   → 交叉验证趋势预测 + 专利白空间
  │
  ├─ 4. 路线图规划（5 分钟）
  │   → Ch20 10 步创新流程：战略→执行
  │   → Ch21 客户收益分析（I-S 矩阵）
  │   → 专利布局策略（Ch22 Implementation + Ch04 Trends）
  │
  └─ 5. 输出
      → 技术路线图报告
      → PPT for 管理层汇报
```

**总耗时**：约 25 分钟
**涉及组件**：SKILL.md → Ch04 → Ch05 → Ch12 → patterns.md → Ch20 → Ch21 → Ch22 → Triz-AI

### 路线 C：极难问题突破（ARIZ）

**适用场景**：你遇到了一个所有人都束手无策的难题，其他方法都失败了。

```
疑难问题
  │
  ├─ 1. 问题准备（5 分钟）
  │   → Ch09 IFR：理想状态是什么？
  │   → Ch12 功能分析：彻底理清系统
  │   → Ch07 心理惯性突破：确认没有被思维定势困住
  │   → Ch08 9-Windows：从时间/尺度维度重新审视
  │
  ├─ 2. ARIZ 执行（15 分钟）
  │   → Step 1: Mini-Problem 定义（最小化改动目标）
  │   → Step 2: Contradiction 深化（深挖矛盾根源）
  │   → Step 3: Resources + ZoF（资源盘点 + 操作区定义）
  │   → Step 4: Solution Search（方案搜索，利用 Ch06 Effects Database）
  │   → Step 5: Refinement（方案评估与精炼）
  │   → 每个步骤的具体检查表见 patterns.md Part 4
  │
  ├─ 3. 验证（3 分钟）
  │   → Ch18 AFI 验证方案是否会失效
  │   → Ch05 理想度评估改进幅度
  │
  └─ 4. 输出
      → ARIZ 完整分析报告
```

**总耗时**：约 30 分钟
**涉及组件**：Ch09 → Ch12 → Ch07 → Ch08 → Ch17 → Ch06 → patterns.md → Ch18 → Ch05

### 路线 D：降本与简化

**适用场景**：产品太贵/太复杂，需要在不牺牲功能的前提下降低成本。

```
降本需求
  │
  ├─ 1. 功能建模（5 分钟）
  │   → Ch12 功能分析：列出所有组件及其功能
  │   → 标注每个组件的功能类型：有用/有害/中性/不足/过度
  │   → 绘制功能-交互关系图
  │
  ├─ 2. 裁剪（5 分钟）
  │   → Ch14 裁剪规则：
  │     - Rule A: 如果组件的功能可以被相邻组件接管 → 删除该组件
  │     - Rule B: 如果组件的功能对用户无价值 → 删除该组件
  │     - Rule C: 如果组件的功能可以由资源提供 → 删除该组件
  │   → 逐组件评估可裁剪性
  │
  ├─ 3. 理想度评估（3 分钟）
  │   → Ch05: 裁剪后的 Ideality = Benefits / (Costs + Harms)
  │   → 确认 Benefits 没有下降，Costs 有显著下降
  │
  ├─ 4. 新矛盾解决（3 分钟）
  │   → 裁剪可能引入新的矛盾
  │   → 回到路线 A 处理新矛盾
```

### 6.5 Step 3: 矛盾矩阵查询（cheatsheet.md → patterns.md）

主要矛盾 参数 21 (Power) vs 参数 22 (Loss of Energy)：

展开每一条（patterns.md）：

**原则 35: Parameter Changes (参数变化)**
- 改变物理状态：相变材料 (PCM) 替代纯液冷 → 利用潜热吸热
- 改变浓度：纳米流体 (nanofluid) 替代纯油 → 导热系数提升
- 改变柔性：柔性导热垫替代刚性冷却结构

**原则 19: Periodic Action (周期性动作)**
- 脉冲冷却：只在温度超过阈值时喷油
- 变频：低速用被动散热，高速切换主动冷却
- 利用暂停：再生制动时反向冷却

**原则 2: Taking Out (提取/分离)**
- 只冷却热点：端部绕组（温度最高处）独立冷却回路
- 干扰分离：将热源（铜损）与热敏感部件（永磁体）热隔离
- 功能分离：结构支撑与散热功能解耦

**原则 15: Dynamism (动态化)**
- 可变冷却：根据工况调节冷却强度
- 可拆分：分段冷却，每段独立控制
- 自适应：油嘴角度随温度自动调节

## 8. 常见错误与规避

### 8.1 触发相关错误

### 9.2 分离原理 (Separation Principles)

| 项目 | 说明 |
|------|------|
| **中文名** | 分离原理（四大分离） |
| **是什么** | 解决**物理矛盾**的核心工具——物理矛盾是"同一个参数既需要大又需要小"的场景，矛盾矩阵无法直接处理，必须用分离原理 |
| **怎么工作** | 四种分离策略：1) **时间分离**（不同时间满足对立需求→活动机翼白天展开夜间折叠）；2) **空间分离**（不同位置满足对立需求→自行车胎面硬耐磨、胎体软吸震）；3) **条件分离**（不同条件下满足对立需求→手术缝合线体内可溶、体外结实）；4) **系统级分离**（不同系统层级满足对立需求→整体飞机刚性+局部蒙皮柔性变形） |
| **何时使用** | 矛盾表述为"X 需要是 A 同时也是非 A"（单一参数的对立需求）；矛盾矩阵返回了"看似矛盾"的建议时 |
| **产出什么** | 一个或多个具体的分离方案（时间/空间/条件/系统级） |
| **关联工具** | 矛盾矩阵（技术矛盾用矩阵）、IFR（理想化的分离：功能存在而系统不存在）、40 发明原则中与分离思想相关的原则 |
| **章节** | Ch04 第 3-5 节（条件/空间/时间分离 + 物理矛盾消解策略） |
| **速查** | patterns.md → ARIZ 4 大分离策略 |

### 9.3 理想度方程 & 理想最终结果 (Ideality Equation & IFR)

| 项目 | 说明 |
|------|------|
| **中文名** | 理想度方程 & 理想最终结果（IFR） |
| **是什么** | TRIZ 的"北极星"——所有解决方案最终都朝着**提高理想度**的方向进化。IFR 是问题的终极状态：系统自己执行所需功能而不存在物理约束 |
| **核心公式** | `Ideality = Σ Benefits / (Σ Costs + Σ Harms)` —— 增大分子（增加有用功能）、减小分母（降低成本 + 消除有害功能） |
| **怎么工作** | 1) 定义当前系统的 Benefits（所有有用功能列表）；2) 列出 Costs（经济、时间、能源、空间）和 Harms（副作用、污染、复杂性）；3) 对每个变法评估理想度变化 → 选择理想度提升最大的方向；4) IFR 版本：问自己"如果系统不存在，如何让功能自己发生？" |
| **何时使用** | 评估多个方案时作为决策判据；所有 TRIZ 问题分析的最后一环（"这个方案真的提高了理想度吗？"）；寻找突破性思路时用 IFR 思维（"这个功能怎么能自己实现？"） |
| **产出什么** | 方案的量化或定性理想度比较；IFR 启发式（"功能怎么不需要系统就能实现？"） |
| **关联工具** | 功能分析（定义 Benefits/Harms 的来源）、裁剪（通过消除组件来降低 Costs + Harms）、S-Curve（成熟期理想度趋于极限） |
| **章节** | Ch05（理想度：定义、公式、三步实现指南、IFR 思维） |
| **速查** | cheatsheet.md → The Ideality Equation 部分 |

### 9.4 功能分析 (Function Analysis)

| 项目 | 说明 |
|------|------|
| **中文名** | 功能分析（组件分析 + 功能建模） |
| **是什么** | 将系统拆解为"组件 → 功能 → 相互关系"的结构化分析方法。它是诊断阶段的基础工具——在解决问题之前先理解问题 |
| **怎么工作** | 1) **组件分析**：列出系统所有组件（超系统、系统、子系统）；2) **功能建模**：用箭头标注组件间的功能交互（有用/有害/不足/过度）；3) 识别：功能缺陷（不足/过剩）、有害作用、缺失的有用功能、过度复杂组件；4) 用"主语-动作-宾语"格式精确描述每个功能 |
| **何时使用** | 问题边界不清晰时（作为分析起点）；产品降本、简化设计；任何 TRIZ 问题建议优先做功能分析 |

### 9.14 预期失效识别 (AFI)

| 项目 | 说明 |
|------|------|
| **中文名** | 预期失效识别（AFI / Anticipatory Failure Identification） |
| **是什么** | TRIZ 的"逆向武器"——不是问"怎么做才能成功？"而是问"怎么做才能让这个东西彻底失败？"然后反过来找到真正的弱点 |
| **核心公式** | AFI Process: **Invert Task → Exaggerate Faults → Generate Faults with TRIZ Tools → Re-invert & Find Fixes** |
| **怎么工作** | 1) **反转任务**：把"如何保证系统可靠性？"→"如何让系统在任意时刻以最坏方式失效？"；2) **夸大故障**：把故障放大到最极端（"不仅泄漏，而要瞬间爆炸"）；3) **用 TRIZ 工具生成故障**：把"造成灾难性失效"当成目标，用矛盾矩阵+物场分析+效应数据库找出所有可能的失效路径；4) **反反转 → 找修复**：回到正常视角，为每条失效路径设计预防措施；5) **排名**：按失效严重度×发生概率排优先级 |
| **何时使用** | 关键安全系统设计（航空、医疗、汽车制动）；FMEA/HARA 的前置补充（找 FMEA 想不到的失效模式）；新方案在实施前的"魔鬼测试" |
| **产出什么** | 所有可能的失效路径清单；每条路径的预防/缓解措施；含严重度×概率的优先级排序 |
| **关联工具** | 矛盾矩阵（生成故障方式）、物场分析（建模失效的物场链）、IFR（有没有失效方案无限接近 IFR？——那就是最危险的失效模式） |

| 5 | 合并/通用化 | 哪些产品线可以合并？ | 过于通用是否失去差异化？ |

| 7 | 嵌套 | 产品/服务如何层层嵌套升级？ | 嵌套是否让用户感到被绑定？ |
| 8 | 配重 | 引入什么"配重"抵消风险？ | 配重本身是否成为新负担？ |
| 9 | 预反作用 | 提前准备什么对冲策略？ | 对冲成本是否超过风险本身？ |
| 10 | 预操作 | 提前建立什么能力/资源？ | 提前投入是否会过早锁定方向？ |
| 11 | 缓冲/保护 | 在价值链何处建立缓冲？ | 缓冲是否降低了效率/敏捷性？ |
| 12 | 等势 | 如何降低客户的更换门槛？ | 低门槛是否降低了客户忠诚度？ |

### 9.16 重要性-满意度矩阵 (I-S Matrix)

| 项目 | 说明 |
|------|------|
| **中文名** | 重要性-满意度矩阵（Importance-Satisfaction Matrix） |
| **是什么** | 从"客户有多在意这个功能 × 当前解决方案有多满意"两个维度评估所有客户需求，将需求分为 4 类并给出对应策略 |
| **怎么工作** | 1) 列出所有客户需求（至少 30-50 条）；2) 对每条需求打分：重要性（1-10）+ 满意度（1-10）；3) 在 2×2 矩阵中定位每条需求：象限 I（高重要性×低满意度→"必须解决"的红色地带）、象限 II（高重要性×高满意度→"保持优势"的绿色地带）、象限 III（低重要性×低满意度→"忽略/延后"）、象限 IV（低重要性×高满意度→"过度服务→砍掉"） |
| **何时使用** | 产品定义阶段（需求优先级排序）；竞品对标（对手在哪个象限？）；资源分配决策；商业模式创新（发现被过度服务的需求→降低标准可降低成本，发现被忽视的必须需求→蓝海机会） |
| **产出什么** | 需求定位矩阵图；优先级排序的改进清单；"过度服务" → 降本机会；"必须解决" → 创新机会 |
| **关联工具** | 12 商业 TRIZ 双重原则（在象限 I 的问题上应用）、功能分析（映射每个需求到具体的功能） |
| **章节** | Ch21（I-S 矩阵：客户需求优先级映射） |

### 9.20 TRIZ 五大哲学支柱 (5 TRIZ Philosophy Pillars)

| 项目 | 说明 |
|------|------|
| **中文名** | TRIZ 五大哲学支柱 |
| **是什么** | TRIZ 所有工具和方法的底层哲学基础——理解了这 5 条，才算真正理解 TRIZ 的"世界观"。这 5 条不是操作工具，而是判断"一个方案是否真正符合 TRIZ 思想"的标准 |
| **五大支柱** | 1) **系统性思维**：一切皆系统，系统有层级（超系统/系统/子系统），问题不在组件本身而在组件间的交互；2) **理想度导向**：所有努力的终极方向是提高理想度 `Ideality = Σ Benefits / (Σ Costs + Σ Harms)`；3) **矛盾驱动**：创新源于矛盾的识别和解决，妥协不是 TRIZ——消除矛盾才是；4) **功能导向**：关注"系统做了什么"而不是"系统由什么制成"——功能可以被不同物理原理实现（功能载体可变，功能不变）；5) **进化规律**：技术系统不随机变异——它们遵循可预测的演化规律（S-Curve + 8 Trends） |
| **何时使用** | 判断方案的 TRIZ 纯度（检查是否符合 5 大支柱）；从战术 TRIZ → 战略 TRIZ（从会用工具到拥有 TRIZ 思维方式） |
| **产出什么** | 思维方式的转变——从"解决这个问题"到"系统性地消除所有问题" |
| **关联工具** | 每个 TRIZ 工具都至少对应 1 条哲学支柱——它们是所有章节的暗线 |
| **章节** | Ch02（5 大 TRIZ 哲学支柱）、Ch06（从组成到功能：功能取向的思考） |
